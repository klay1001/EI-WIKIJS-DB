Overview

本文件記錄同質(Homogeneous)系統複製 (System Copy)過程的操作。來源系統和目標系統都是 Sybase ASE + SUSE Linux 的組合，二個系統都已經存在。本例採用的資料複製方法是 Sybase Backup/Restore，也就是拿原始系統的 dump file 在目標系統 load 回去。整個過程沒有用到 SWPM, 也不用重裝 SAP Application 或 Sybase database instance。絕大部份的動作是利用 isql 及 Linux command 完成。

本例的來源系統 Sybase ASE 的 Database ID 是 PRD,  目標系統 Database ID 是 QAS。且 二個資料庫的 sapsa 和 SAPSR3 帳號之密碼不相同。Sybase ASE 的版本是 16.0.2

Step 2: 匯出 Source System Sybase ASE 的 DDL

SOURCE SYSTEM

以 syb<sid> 登入到 Source System，進入 isql
註: 下圖的 myisql 為自撰的 script，含呼叫 QAS isql 必要參數，執行時進入 isql 環境。
若直接呼叫 isql，請用  isql -Usapsa -X -D<DB ID>  按 Enter 後打入 sapsa 密碼可進入 isql。

切換到 Source System database (本例為 PRD)，執行 sp_ddlgen 匯出 PRD 的 DDL。

2> go
3> sp_ddlgen ‘database’, ‘PRD’

4> go > DDL_PRD.sql

執行後會把產出的 SQL statement 存到“DDL_PRD.sql” 檔案裡。

DDL 檔案描述重建一個 PRD 一樣的 Database，所需要的 data & log devices, 以及他們的大小。

但 DDL 不會建立這些 data & log devices。它假設先有這些 devices 存並有足夠的空間，而 DDL 是把必要的空間大小配給 PRD database。
之後我們會利用這個 DDL的內容在目標系統重建一個和 PRD database 大小一樣(或更大的) QAS 資料庫。

Data & log device 清掉, 如 :

2>    go

2) 在 Drop database 之前，在 QAS 上執行 sp_ddlgen 把 QAS 自己的 device 定義匯出參考

Step13: 執行 DDL 建立 QAS database

TARGET SYSTEM

在執行 DDL_QAS.sql 之前，請不要忘了把上一步中有改變 device size 大小更新到 DDL_QAS.sql 裡。執行 DDL_QAS.sql 的動作，不用進 isql 環境裡面 ，以 syb<sid> 身份在 linux command line 執行方式為 :
isql -Usapsa -P<password> -X -S <DBSID> -i <Script file name>   >   <output log file name>
本例的指令是
isql -Usapsa -P******* -X -S QAS -i DDL_QAS.log > DDL_QAS.log

右方 screen 為執行時監看 DDL_QAS.log 內容

最後執行完時，isql command 會回到 linux promopt, log 檔裡最後顯示 100% (^C 跳出它)

Step 14: 補足不足的 data/log file (Optional)

TARGET SYSTEM

接下來要 load backup file, 請在 isql 環境裡，一樣在 master db，執行 :

1>    load database <DBSID> from ‘<path to the backup dump file>’

2>    go

請注意，如果一切順利沒有跳錯，請直接到Step15。

 

若此時你所開的 database 還不夠大，load database 會中斷並告知你要多少空間。如下圖，因為 log device 只有 20420MB，比原本要的 20480MB 少 (見 Step 12), 所以還是要加回去。但不用加在同一個 device，只要總共的 log device 到 20480MB 就好。

 

下圖是加一個 1GB QAS_log_002 上去 (留意 ALTER DATABASE QAS LOG ON�)

 

同樣的 data device 也有不足，下圖裡我們再加上二個 8GB QAS_data_019, QAS_data_020 上去

Step 15: load database à 完成 load

TARGET SYSTEM

如果 load database 下去後，停在下面的畫面至少 5 秒(可能幾分鐘)，表示 load database 沒有檢查到空門是不足，正在進行 loading，請耐心等待。

等一陣子後，出現百分比 (n%) LOADED.

等到 100% LOADED，還會有一些後續動作

直到 isql prompt 回來

Step 16:  online database

TARGET SYSTEM

Load database 結束後，不代表 QAS database 已啟動，此時應可以再load transaction 來補 transaction log file (類似 oracle 的 recover database �..), 確定沒有再補什麼檔後，在 isql 下

1>    online database QAS

2>    go

Step 17: Start SAP Application

TARGET SYSTEM

在啟動 sap 之前，為了怕 background job 跑起來 (原系統在備份時沒有下
BTCTRNS1)，所以請手動加一行(如下)以防止 BTC job schedule 計時工作。
  disp/btctime = 0

到 DEFAULT.PFL 裡，然後再啟動 SAP Application。

此時應可用 SAP GUI 登入系統了

Step 18: DO post-copy activities

TARGET SYSTEM

登入 SAP GUI ，以下幾件事

1)     T-Code: SE38 à BTCTRNS1

2)     T-Code: SICK

3)     Stop SAP, 拿掉 disp/btctime = 0, Start SAP

4)     T-Code: SLICENSE, 更新SAP License

5)     T-Code: RZ10 import profile 並修改 logon client 為 888

6)     T-Code: STMS (應會先動抓到 TMS Domain Controller 在DEV)

7)     到 DEV 000 再做一次 TMS update到各系統，應可把 QAS 加到 TMS route 裡。

8)     T-Code: SCC4，把 client 由 production 改為 test (不要改 logical system)

9)     T-Code: BDLS ，把 PRDCLNT888 都改為 QASCLNT888 (run in background)

10)  T-Code: SE61 , 修改 logon screen 的文字，拿掉 production 字樣。

11)  備份

12)  大功告成 !!!
